## 📁 Namespaces

> **Namespace** provides logical isolation of all K8s resources for a particular application.

By default, all resources in Kubernetes are created in the **`default`** namespace.

### Default Namespaces

| Namespace | Purpose |
|-----------|---------|
| `default` | Where pods are created if no namespace is specified |
| `kube-node-lease` | Node heartbeat leases |
| `kube-public` | Publicly readable resources |
| `kube-system` | System-level K8s components |

---

### 🔧 Namespace Commands

```bash
# Create a namespace
kubectl create ns <namespace>

# List all namespaces
kubectl get ns

# Check current namespace
kubectl get pods
kubectl config view        # Shows nothing if in default namespace
```

### ✅ Set & Switch Namespace

```bash
# Switch to a namespace
kubectl config set-context --current --namespace=dev

# Verify
kubectl config view
```

### 🔍 Cross-Namespace Operations

```bash
# View pods in another namespace (from any namespace)
kubectl get pods -n <namespace>
kubectl get pods -n dev

# View services in another namespace
kubectl get svc -n dev
kubectl get svc -n test

# Delete a pod in another namespace (⚠️ not recommended — use RBAC)
kubectl delete pod dev-pod1 -n dev
```

### 🗑️ Delete a Namespace

> ⚠️ This deletes **everything** inside the namespace!

```bash
kubectl delete ns <namespace>
```

### 📄 Namespace Manifest Example — `nameswiggi.yaml`

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: swiggi-app
---
apiVersion: v1
kind: Pod
metadata:
  name: swiggi-app-pod
  namespace: swiggi-app
  labels:
    app: swiggi-app
spec:
  containers:
    - name: swiggi-app-container
      image: kastrov/swiggi
      ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: swiggi-app-service
  namespace: swiggi-app
spec:
  selector:
    app: swiggi-app
  type: NodePort
  ports:
    - protocol: TCP
      port: 80         # Service port
      targetPort: 80   # Container port
      nodePort: 30080
```

```bash
kubectl apply -f nameswiggi.yaml
kubectl get svc -n swiggi-app
```

## 🎮 Kubernetes Controllers

> A K8s **controller** maintains the desired number of pods for HA (High Availability).

```
Desired State  →  10 pods
Actual State   →  10 pods  (if 5 crash → controller recreates 5)
```

If a pod **crashes or is deleted**, the controller notices and **recreates** it — this is called **reconciliation**.

### Controller Types

| Controller | Description |
|------------|-------------|
| **Replication Controller (RC)** | Original, basic replica management |
| **Replica Set (RS)** | Advanced RC with multi-label support |
| **Deployment** | RS + rolling updates + rollback |
| **DaemonSet** | One pod per node |
| **StatefulSet** | Ordered, stateful pods |

---

## 🔁 Replication Controller (RC)

> Ensures a specified number of pod replicas are running at any given time. If a pod fails or is deleted, RC creates a new one.

### 📄 Manifest — `02-rc.yaml`

```yaml
apiVersion: v1
kind: ReplicationController
metadata:
  name: zomato-rc
spec:
  replicas: 2
  selector:
    app: zomato-app       # Must match pod labels
  template:
    metadata:
      name: zomato-pod
      labels:
        app: zomato-app
    spec:
      containers:
        - name: zomato-container
          image: kastrov/zomato
          ports:
            - containerPort: 3000
---
apiVersion: v1
kind: Service
metadata:
  name: zomato-service
spec:
  type: LoadBalancer
  selector:
    app: zomato-app
  ports:
    - port: 80
      targetPort: 3000
```

```bash
kubectl apply -f 02-rc.yaml
kubectl get pods
kubectl get rc
kubectl get svc

# Manual Scaling
kubectl scale rc zomato-rc --replicas 6
kubectl get pods
```

### ⚠️ RC Limitations

- ❌ Does **not** support multiple labels on pods (equality-based selector only)
- ❌ Rolling updates are **not** possible (very slow)
- ❌ **Deprecated** — use ReplicaSet or Deployment instead

**Homework:** Create a ReplicationController inside a specific namespace.

## 🔂 Replica Set (RS)

> Advanced version of RC. Supports **multi-label selectors** (`matchLabels`).

### RC vs RS Selectors

```yaml
# RC — equality-based (only one label)
selector:
  app: zomato-app

# RS — set-based (multiple labels)
selector:
  matchLabels:
    app: zomato-app
    version: v2
    env: prod
```

### 📄 Manifest — `03-rs.yaml`

```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: zomato-rs
spec:
  replicas: 3
  selector:
    matchLabels:
      app: zomato-app
  template:
    metadata:
      name: zomato-pod
      labels:
        app: zomato-app
    spec:
      containers:
        - name: zomato-container
          image: kastrov/zomato
          ports:
            - containerPort: 3000
---
apiVersion: v1
kind: Service
metadata:
  name: zomato-service
spec:
  type: NodePort
  selector:
    app: zomato-app
  ports:
    - port: 80
      targetPort: 3000
      nodePort: 30000
```

```bash
kubectl apply -f 03-rs.yaml
kubectl get pods
kubectl get rs
kubectl get svc

# View labels
kubectl get pods --show-labels

# Scaling
kubectl scale rs/zomato-rs --replicas=10
kubectl scale rs/zomato-rs --replicas=3

# Cleanup
kubectl delete -f 03-rs.yaml
```

### ⚠️ RS Limitation

> Changing the image in RS **does NOT restart existing pods** — only new pods get the new image.
> **Rolling updates / Rollouts are NOT possible with RC or RS.**

## 🚀 Deployment

> Deployment = ReplicaSet + **Rolling Updates** + **Rollback** support.
> Use deployments to manage **multiple versions** of the same application.

### Deployment Strategies

| Strategy | Description |
|----------|-------------|
| **Rolling Update** *(default)* | Delete one old pod → create one new pod, gradually |
| **Recreate** | Delete ALL old pods → create all new pods (downtime!) |
| **Blue-Green** | Run both versions simultaneously, switch traffic |
| **Canary** | Gradually shift traffic to new version |

---

### 📄 Manifest — `04-deployment.yaml`

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: zomato-deployment
  labels:
    app: food
spec:
  replicas: 3
  selector:
    matchLabels:
      app: food
  template:
    metadata:
      labels:
        app: food
    spec:
      containers:
        - name: zomato-container
          image: kastrov/zomato
---
apiVersion: v1
kind: Service
metadata:
  name: zomato-service
spec:
  type: LoadBalancer
  selector:
    app: food
  ports:
    - port: 80
      targetPort: 3000
```

```bash
kubectl apply -f 04-deployment.yaml
kubectl get deployments
kubectl get pods

# Scaling
kubectl scale deploy/zomato-deployment --replicas=10
kubectl scale deploy/zomato-deployment --replicas=3

# Check running image
kubectl describe pod -l app=food | grep -i image
```

### 🔄 Rolling Update (Image Update)

```bash
# Edit deployment (do NOT edit the yaml file directly in production)
kubectl edit deploy/zomato-deployment
# Change: image: kastrov/zomato  →  image: kastrov/swiggy
```

### 📜 Rollout History & Rollback

```bash
# View rollout history
kubectl rollout history deploy/zomato-deployment

# Add annotation/description to a revision
kubectl annotate deployment zomato-deployment \
  kubernetes.io/change-cause="Updated zomato to swiggy" --overwrite

# Check rollout status
kubectl rollout status deploy/zomato-deployment

# Rollback to previous version
kubectl rollout undo deploy/zomato-deployment
kubectl get pods
```

**Example rollout history:**
```
REVISION  CHANGE-CAUSE
1         <none>
2         <none>
3         Updated zomato to swiggy
```

## 🔵🟢 Blue-Green Deployment

> **Blue** = current live version (users accessing it)
> **Green** = new version (deployed, but not yet serving traffic)
>
> Traffic switches **immediately** by updating the service selector.

```
  Blue  (v1) ──────┐
                    ├──▶  Common LoadBalancer  ──▶  Users
  Green (v2) ──────┘
         ↑
    (switch selector here)
```

---

### 📄 `blue-deploy.yaml`

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-blue
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
      version: blue
  template:
    metadata:
      labels:
        app: nginx
        version: blue
    spec:
      containers:
        - name: nginx
          image: nginx
          ports:
            - containerPort: 80
          command: ["/bin/sh", "-c"]
          args:
            - |
              echo '<html><body style="background-color:skyblue;">
              <h2>BLUE (Version 1)</h2>
              </body></html>' > /usr/share/nginx/html/index.html && nginx -g "daemon off;"
          readinessProbe:
            httpGet:
              path: /
              port: 80
            initialDelaySeconds: 5
            periodSeconds: 5
```

### 📄 `green-deploy.yaml`

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-green
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
      version: green
  template:
    metadata:
      labels:
        app: nginx
        version: green
    spec:
      containers:
        - name: nginx
          image: nginx
          ports:
            - containerPort: 80
          command: ["/bin/sh", "-c"]
          args:
            - |
              echo '<html><body style="background-color:lightgreen;">
              <h2>GREEN (Version 2)</h2>
              </body></html>' > /usr/share/nginx/html/index.html && nginx -g "daemon off;"
          readinessProbe:
            httpGet:
              path: /
              port: 80
            initialDelaySeconds: 5
            periodSeconds: 5
```

### 📄 `service.yaml` (Common Service)

```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-lb
spec:
  type: LoadBalancer
  selector:
    app: nginx
    version: blue   # ← Change to "green" to switch traffic
  ports:
    - port: 80
      targetPort: 80
```

### 🚀 Deploy & Verify

```bash
kubectl apply -f blue-deploy.yaml
kubectl apply -f green-deploy.yaml
kubectl apply -f service.yaml

kubectl get pods
kubectl get svc
kubectl get endpoints nginx-lb
```

### 👁️ Live Traffic Watch

```bash
# watch — runs a command repeatedly every N seconds
# -n 1  — every 1 second

watch -n 1 curl http://<LB-URL>
watch -n 1 curl http://<IP-URL>
```

### 🔀 Switch Traffic to Green

```bash
# Edit service.yaml: change version: blue  →  version: green
kubectl apply -f service.yaml

# Or patch directly
kubectl patch svc nginx-lb -p '{"spec":{"selector":{"app":"nginx","version":"green"}}}'
```
